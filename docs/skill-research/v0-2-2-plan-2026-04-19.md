# Plan — skill-forge v0.2.2 ("system-aware Phase 1")

Target version: **v0.2.2**. Trigger: Q-SYS dry-run (2026-04-19) surfaced 12 gaps, all centred on the same root cause — Phase 1 only inventories `~/.claude/skills/`, ignoring the rest of the developer surface (hooks, MCPs, KBs, non-JS/Python stacks, monorepos, project-local skills).

One theme, one release. Same philosophy as v0.2.1: process fixes over new models.

## Staged rollout

### Stage 1 — TR004 audit rule (≈45 min)

**Goal:** audit detects skills with no filePattern AND no bashPattern → WARNING, unless a hook covers the same file pattern (in which case INFO).

**Files changed:**
- `skill-forge/scripts/audit.sh`

**Change detail:**
1. Add `TR004` to the rule-ID comment block (line 16 region)
2. After the existing TR001/TR002/TR003 block, add a new check inside the per-skill Python loop (around line 259):
   ```python
   # TR004: skill has no triggers (description-only)
   if not file_patterns and not bash_patterns:
       emit "TR004" "WARNING" "Skill declares no filePattern or bashPattern — discoverable via description matching only. Add triggers or justify in main SKILL.md."
   ```
3. Refinement path: if we also inventory hooks in Phase 1 (Stage 2), extend TR004 to soften the warning when a PostToolUse hook matches a file type plausibly associated with the skill's topic. Stage 1 ships the basic rule; the hook-softening is additive.

**Test:**
- Run `audit.sh ~/.claude/skills/qsys-plugin-patterns` — expect TR004 WARNING (currently has no triggers)
- Run against `~/.claude/skills/skill-forge` — expect clean (has `filePattern: []` + `bashPattern: []` declared; that's "declared empty" not "absent" — TR004 should ignore empty-declared as an explicit choice)

**Independent of other stages. Ship first.**

### Stage 2 — Phase 1 inventory expansion (≈90 min, biggest stage)

**Goal:** `profile.json` reflects the whole developer surface, not just the skill library.

**Files changed:**
- `skill-forge/references/phase-1-discover.md` (primary)
- `skill-forge/skill-forge/SKILL.md` (phase-1 summary row)

**Schema additions to profile.json:**

```json
{
  // ... existing fields ...
  "active_hooks": [
    {"event": "PostToolUse", "matcher": "Write|Edit", "command": "bash ~/.claude/hooks/qplug-validate.sh"},
    {"event": "UserPromptSubmit", "matcher": "*", "command": "bash ~/.claude/hooks/qsys-kb-search.sh"}
  ],
  "available_mcp_servers": ["jcodemunch", "jdocmunch", "open-brain"],
  "local_knowledge_bases": [
    {"path": "~/ai/qsys-kb", "hint": "Neo4j + Qdrant, Q-SYS content"},
    {"path": "~/ai/knowledge-base", "hint": "general KB"}
  ],
  "project_local_skills": [
    {"path": "<project>/.claude/skills/qsys-plugin-development", "name": "qsys-plugin-development"}
  ],
  "sub_projects": [
    {"path": "cisco-modular-plugin", "languages": ["lua"], "stack_tags": ["qsys-plugin", "cisco-roomos"]},
    {"path": "qpdk", "languages": ["python"], "stack_tags": ["qsys-tooling", "lua-embed"]}
  ]
}
```

`sub_projects` is populated only when depth-1 scan finds ≥2 manifests and root has none.

**New sub-steps to add to `phase-1-discover.md`:**

- **§1.4a Inventory hooks** — read `~/.claude/settings.json` (jq for keys only, never values); extract `.hooks.{PostToolUse,UserPromptSubmit,SessionStart,Stop}[].hooks[]` into `active_hooks`.
- **§1.4b Inventory MCP servers** — read `~/.claude.json` with `jq -r '.mcpServers | keys[]'` (names only; values contain API keys — never read full).
- **§1.4c Inventory local knowledge bases** — glob `~/ai/*knowledge*`, `~/ai/*kb*`, `~/ai/*ingestion*`. For each, check if there's a `hook` that references it; record path + optional hint from first README line.
- **§1.4d Inventory project-local skills** — glob `<project>/.claude/skills/*/SKILL.md` and `<project>/*/.claude/skills/*/SKILL.md`. Parse frontmatter same as §1.3.
- **§1.1a Monorepo detection** — if root has no `CLAUDE.md`/`README.md`/`package.json`/`pyproject.toml` but `.git/` exists, run depth-1 manifest scan. If ≥2 sub-projects found, `AskUserQuestion`:
  > Options: `Target whole monorepo` / `Target one sub-project` / `Stop`
- **§1.5 language dispatch** — Phase 1.5 grep recipe becomes a dispatch:
  - JS/TS: existing `jq` + `package.json` path
  - Python: existing `pip list` path
  - **Lua/Q-SYS**: `grep -rEoh "^(Controls|Component|Timer|TcpSocket|SSH|HttpClient|Design|System|Socket)\\." + "^\\s*require\\s*\\(?[\"'][^\"']+[\"']"` → top namespaces become `heavy_api_surface`
  - Fallback: if no manifest recognised, skip grep and note `profile_completeness.grep_skipped_reason: "unrecognised_stack"`

**Phase 1.3 relevance algorithm update** — skills are relevant if ANY of:
1. `filePattern` glob matches ≥1 file in target (existing)
2. `bashPattern` appears in project scripts/history (existing)
3. **NEW:** description keywords overlap ≥2 `tech_stack_tags` or `heavy_api_surface` namespaces
4. **NEW:** an active hook covers a file pattern related to the skill's topic (soft signal; note but don't auto-match)

**Test:**
- Re-run Phase 1 on `~/ai/qsys-plugins`. Expect:
  - `active_hooks` lists qplug-validate + qsys-kb-search
  - `available_mcp_servers` includes jcodemunch, jdocmunch, open-brain
  - `local_knowledge_bases` includes ~/ai/qsys-kb
  - `relevant_skills` includes all 4 Q-SYS skills (via keyword fallback, since none declare filePattern)
  - `sub_projects` populated with cisco-modular-plugin, qpdk, qsys-status-gate, etc.

**Depends on:** Stage 1 TR004 (so that Phase 1.3 keyword-fallback has a corresponding audit rule).

## Risk register

**R1 — Phase 1.1 monorepo AskUserQuestion** interrupts autopilot. The stop is justified (you can't silently target a monorepo when each sub-project has a different stack), but worth noting it adds a 6th autopilot stop beyond the documented 4+Phase-9. Mitigation: only fire the question when sparse-root AND ≥2 sub-project manifests found. Otherwise auto-proceed.

**R2 — `settings.json` / `.claude.json` are on the DO-NOT-READ list** per root CLAUDE.md (contain secrets). Phase 1.4a/1.4b must use `jq` for keys only, never dump full file content. Add explicit instruction in the reference text so future agents don't paste full-file reads.

**R3 — `available_mcp_servers` varies per-user.** Don't hardcode names in skill-forge; detect dynamically. The research-agent brief should say *"use whichever of these are available"* rather than prescribing jcodemunch specifically.

**R4 — profile.json schema change breaks `--from-phase=N` with old profiles.** Add a `profile_schema_version` field; on load, if `schema_version < 2`, run a migration that populates new fields with safe defaults (`active_hooks: []` etc.). One-time pain.

**R5 — TR004 false-positives on skills that *intentionally* have no triggers** (e.g., `writing-skills` is reference-only). Allow an escape hatch: frontmatter `triggers_intentionally_absent: true` suppresses TR004. Document sparingly.
